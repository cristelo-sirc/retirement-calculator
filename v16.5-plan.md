# v16.5 Mobile Fix Cycle -- Plan

**Date:** 2026-03-19
**Status:** Approved &mdash; Implementing
**Scope:** Mobile-only UI fixes. Zero engine or calculation changes.

---

## Confirmed Bugs (from screenshots)

### Fix 1 -- Mobile Nav Bar Not Accessible
**Problem:** After the wizard completes (or is skipped), the mobile nav bar disappears or is inaccessible. Users are stranded with no way to navigate between Dashboard, Charts, What-If, and Reports. The FAB button opens the input panel but nothing navigates between tabs.

**Root Cause:** The `mobile-nav-bar` sits in the DOM directly after the top `<nav>` -- it scrolls off the top of the viewport with the page content. It is not `position: fixed`, so it vanishes the moment the user scrolls. Additionally, iOS Safari dynamically hides/shows its chrome, further compressing the visible area. The wizard overlay (z-index: 9999) also fully covers it while active, and no post-wizard logic restores scroll position.

**Fix:** Convert the `mobile-nav-bar` from a top-anchored scroll-away bar to a **fixed bottom tab bar**. This is the standard mobile navigation pattern (think every banking or finance app). It stays anchored to the bottom of the viewport at all times, regardless of scroll position or iOS chrome behavior. Requires `position: fixed; bottom: 0; left: 0; right: 0` plus `padding-bottom: env(safe-area-inset-bottom)` for iPhone home indicator.

**Risk:** Low. CSS-only repositioning. Must ensure `main-content` gets matching `padding-bottom` so content is not hidden behind the fixed bar.

---

### Fix 2 -- Wizard Step Progress Bar (Scrollable, Labels Preserved)
**Problem:** 11 steps with full labels (Start, Timeline, Portfolio, Income, SS, Pension, Spending, Health, Adjust, Invest, Tax) at 375px renders as an illegible, cramped strip. Labels overlap or get cut off.

**Fix:** Make the progress bar **horizontally scrollable** and auto-scroll to keep the active step centered in view on each step change. Labels remain full-size and unchanged. The bar container gets `overflow-x: auto; -webkit-overflow-scrolling: touch` with `scroll-behavior: smooth`. JS auto-centers the active dot via `scrollIntoView({ behavior: 'smooth', inline: 'center' })` when the step advances or retreats.

**Risk:** Low. Scrollable containers are well-supported. Labels and dot logic untouched.

---

### Fix 3 -- Wizard Footer Clipped by iOS Safe Area
**Problem:** The Back / Next button bar at the bottom of the wizard is partially hidden behind iOS Safari's browser chrome (home indicator + tab bar). Buttons are accessible but require the user to know to scroll down.

**Fix:** Add `padding-bottom: max(16px, env(safe-area-inset-bottom))` to `.wizard-footer`. One-line CSS addition that expands the footer padding on notched/home-indicator devices and has no effect on non-iOS devices.

**Risk:** Very low. CSS-only, additive.

## Additional Responsive Issues Found in Code Review

These were not in the screenshots but are latent bugs or near-misses based on code audit.

### Fix 4 -- FAB Button Partially Hidden by iPhone Home Indicator
**Problem:** The mobile FAB (edit inputs button) is positioned `bottom: 24px` with no safe-area awareness. On iPhones with a home indicator (iPhone X and later -- essentially all modern iPhones), the home indicator overlaps the bottom ~34px of the screen. The FAB may be partially hidden or harder to tap.

**Fix:** Change FAB `bottom` to `max(24px, env(safe-area-inset-bottom) + 8px)`.

**Risk:** Very low.

---

### Fix 5 -- Wizard Has Zero Mobile CSS
**Problem:** There are no `@media (max-width: 768px)` rules targeting any wizard class. The wizard was designed for a 680px desktop card. On mobile the card fits (`max-width: 95%`) but the internals do not adapt:
- `.wizard-grid` (2-column) and `.wizard-grid-3` (3-column) remain multi-column, squishing input fields
- `.wizard-body` padding is 24px on all sides (wasteful on a 375px screen)
- `.wizard-header` padding is desktop-sized

**Fix:** Add mobile media query for wizard internals:
- `.wizard-grid` and `.wizard-grid-3` collapse to `grid-template-columns: 1fr` on mobile
- Reduce `.wizard-body` padding to `16px`
- Reduce `.wizard-header` padding to `12px 16px`

**Risk:** Low. Layout only, no logic.

---

### Fix 6 -- Mobile Done Bar Needs Safe-Area Padding
**Problem:** The sticky "Done" bar at the bottom of the mobile input panel uses `position: sticky; bottom: 0`. On iPhone, this may sit behind the home indicator.

**Fix:** Add `padding-bottom: env(safe-area-inset-bottom)` to `.mobile-done-bar`.

**Risk:** Very low.

---

### Fix 7 -- Toast Notification Collides with Fixed Nav Bar
**Problem:** Mobile toast is currently positioned `bottom: 80px` -- a hardcoded offset. Once Fix 1 converts the nav bar to a fixed bottom bar (~56px tall), the toast needs to sit above it or they will overlap. iPhone home indicator compounds this further.

**Fix:** Update toast `bottom` to `calc(56px + max(16px, env(safe-area-inset-bottom)) + 8px)` -- i.e., above the fixed nav bar with margin. Coordinate with Fix 1.

**Risk:** Low. Dependent on Fix 1.

## Refinements Added During Review

### Refinement A -- FAB/Nav Bar Collision (extends Fix 1 + Fix 4)
When the bottom nav bar is fixed, the FAB button must sit *above* it, not overlap. Fix 4 updated to position FAB above the nav bar height + safe area, not just safe area alone.

### Refinement B -- Scrollable Step Bar Fade Hint (extends Fix 2)
Horizontal scroll containers have no visible scrollbar on mobile. Add subtle gradient fade masks on the overflow edges to signal "more content." CSS `mask-image` with linear gradients, toggled by scroll position via JS.

### Refinement C -- Bottom Nav z-index Spec (extends Fix 1)
Explicit z-index layering: bottom nav at `z-index: 1000`, below wizard overlay (9999) and modals, above regular content. Prevents stacking surprises.

### Refinement D -- Main Content Bottom Padding (extends Fix 1)
`padding-bottom` on main content must account for nav bar height *plus* safe-area inset, not just nav bar alone: `calc(56px + env(safe-area-inset-bottom))`.
